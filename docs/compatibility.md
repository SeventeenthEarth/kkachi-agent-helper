## Compatibility rules

- KHS may consume KAH `@latest` when `kkachi-agent-helper capabilities --json` reports the command surfaces and compatibility flags KHS requires. KHS release tags should also publish tested/recommended KAH versions for reproducible historical runs.
- Helper state schemas are versioned by the project-local `.kkachi/config.yaml` and embedded schema registry.
- `project init` owns KAH project bootstrap files: `.kkachi/config.yaml`, `.kkachi/project-overlay.yaml`, `docs/kkachi-docs-map.yaml`, status, events, and schema copies.
- `project init --force` reconfigures bootstrap files without deleting status, events, runs, artifacts, or gate history.
- KAH does not install KHS/Hermes skill content, templates, registries, or evaluation assets. Use Hermes native skill installation for skill packages.
- KHS owns workflow policy: whether KHS should trigger, phase applicability/order, normalized `checklist.md` generation, backend-use decisions, and commander reasoning. KAH owns deterministic state, artifact, schema, gate, event, lock, diagnostics, command-surface, phase-plan, backend-evidence, and approval validation after KHS or the user chooses to apply the workflow.
- KAH's `plan` gate requires completed `acceptance-criteria.md`, `plan.md`, and KHS-normalized `checklist.md`, but it does not parse or require KAB planner-only sections such as `KHS Checklist Seed`.
- KHS declares backend evidence requirements through `run create --backend-evidence auto|required|not_applicable`; KAH stores the resolved `backend_evidence` value, initializes required backend artifacts only when declared required, and validates artifact shape/completion without choosing or overriding the backend.
- KHS may write canonical run artifacts through `artifact write`, `artifact append`, and markdown `artifact set-status` to get KAH path-safety checks, atomic file replacement, and `artifact.written` audit events. Direct artifact file compatibility remains available during migration; this release intentionally rejects unmanaged supplemental artifact paths. Schema-owned backend JSON such as `selected-cli.json` and `bridge-session-snapshot.json` must be written with `artifact write`; generic lifecycle `set-status` is rejected with `artifact_status_not_applicable` so semantic fields like `selected-cli.json.status=supported|degraded` are preserved and `gate check backend` remains the completion validator.
- KHS may consume `project probe-toolchain --json [--project-root <path>]` for TOLMR generated local toolchain state. KAH reports deterministic helper command/version/path and project facts plus `no_write.guaranteed=true` / `write_count=0`; KAS owns `.kkachi/toolchain.yaml` schema, writes, stage policy, MAR/provider policy, legacy import, and interpretation.
- KAS/KHS may consume `kkachi-agent-helper gjc start-deep-interview|start-ralplan|start-ultragoal --run <run_id> --task <task_id> --packet <run-local-packet> --json`, `kkachi-agent-helper gjc status --run <run_id> --json`, `kkachi-agent-helper gjc callback-kanban ... --json`, and `kkachi-agent-helper gjc lock-plan ... --json` only when the effective binary advertises the `gjc` command group and `gjc_evidence_wrapper=true` through `capabilities --json`. `start-ralplan` requires KAS `native_ralplan_input.stage`, `.stage_n`, and `.artifact` fields and adapts them into installed GJC 0.7.3 native `ralplan --write` flags; GAJAE-008 still needs `start-ultragoal` native brief adaptation. Callers must not assume `--packet` alone is passed through to live GJC. KAH normalizes GJC execution to `HOME=/Users/draccoon`, creates/reuses run-local `GJC_SESSION_ID` evidence, records process/status, records `packet_ref` as KAS input packet evidence, records `receipt_ref`, records `artifact_refs` as GJC candidate output evidence, records callback idempotency/no-wake-claim metadata, records KAS-supplied accepted plan hashes after an approval ref, and preserves refs/hashes under `.kkachi/runs/<run_id>/artifacts/gjc/`. KAH treats GJC output, callbacks, and plan locks as candidate/mechanical evidence only. KAH does not decide KAS plan policy, color/MAR/final acceptance, backend selection, KAB activation, or provider/auth/token/gateway/model/profile settings. Missing `gjc`, missing/malformed `native_ralplan_input`, unsafe native ralplan artifacts, missing/malformed session or status evidence, unsafe/cross-run/missing refs, missing/drifted `packet_ref`, malformed hashes, checksum mismatch, stale or mismatched `status_hash`, unsupported statuses/actors, malformed GJC JSON, missing plan hashes for `ralplan_ready`, callback idempotency conflicts, and locked plan drift fail closed; recovery for packet drift is to regenerate or repair the run-local KAS packet before consuming GJC status.
- KHS may store declared workflow state in `.kkachi/runs/<run_id>/phase-plan.yaml` through `phase-plan init/show/set/validate`. KAH validates declared structure, reasons, feedback bounds, final evidence links, and approval records for rows marked `approval_required: true` only; KHS owns phase applicability, ordering policy, and workflow decisions.
- Workflow graph compatibility: `.kkachi-workflow.yaml` plus `kkachi-agent-helper graph init`, `graph validate`, `graph explain`, `graph diff`, `graph propose`, `graph apply`, `graph export`, and diagnostics `graph_compatibility` are supported when the effective KAH binary advertises those graph subcommands and compatibility flags through capabilities/help. KHS chooses templates, policy, declarative complete candidate graph proposals, and approval semantics; KAH initializes from `khs-default` or explicit template paths, validates/explains/diffs graph state, records proposal evidence, applies approved proposal records, exports non-authoritative generated diagrams, and never silently consumes fallback workflow policy. Missing, stale, broken, or otherwise invalid project graph states may be repaired only through approval-gated complete-candidate proposal/apply evidence; KAH records base validation status/reason codes, fail-closed drift checks, optional backup/recovery refs, and `graph.applied` events. `graph validate --json`, `graph explain --json`, proposal endpoint evidence, and diagnostics `graph_compatibility` expose stable `reason_codes` for implemented graph states so KAS can distinguish missing, valid, invalid-schema, parse-error, source-precedence, feedback-intake, repairability, and nested forbidden-fallback facts without KAH deciding KAS policy. Checksum/audit, manual-edit, phase-conflict, and approval-required codes are reserved vocabulary for producer-backed states and must not be treated as emitted until corresponding validation/diagnostic evidence exists. KHS must not silently edit `.kkachi-workflow.yaml` as fallback when graph apply support is missing.
- Workflow catalog promotion compatibility: `kkachi-agent-helper workflow catalog propose/apply` is supported only when the effective KAH binary advertises `workflow_catalog_proposal_apply=true` and workflow subcommands `catalog propose` and `catalog apply` through capabilities/help. KAS WFLOW-009 supplies complete candidate content and approval evidence; KAH validates candidate workflow DAG/catalog/node-contract content, records proposal evidence, enforces base/candidate checksum drift checks, verifies hash-bound approval, creates backups for existing targets, writes project-local workflow catalog targets, and appends audit events. KAH must not decide promotion policy, workflow semantic preference, selector/ranking behavior, trigger generation policy, agent assignment, backend fallback/execution, retry/rollback automation, dynamic node creation, or automatic promotion. Missing support in the installed/effective binary means KAS promotion apply must fail closed rather than direct-write `.kkachi/` workflow state.
- Declarative workflow gates are supported as graph extensions to built-in KAH gates: built-in gates win name resolution, graph-declared gates with fixed check types may be run through `gate check`, and graph gates with `final_required: true` are required and freshness-checked by `gate final`. - Graph source precedence must fail closed: `.kkachi/config.yaml`, generated Mermaid/PlantUML diagrams, stale `.kkachi/` runtime state, KHS defaults, and Kkachi v2 `.kkachi/config/workflows/` are never fallback graph authority. ## Capabilities schema evolution

`capabilities_schema_version` versions the `capabilities --json` contract independently from helper release version and project state schema version.

- Additive changes that preserve existing field names, JSON types, and meanings may keep the same capabilities schema version. Examples: adding a new command group, subcommand, compatibility flag, or deprecated/omitted surface entry.
- Breaking changes must bump `capabilities_schema_version`. Examples: removing, renaming, or changing the JSON type of an existing field; changing the meaning of an existing compatibility flag; changing a previously supported command surface to unsupported without first exposing a deprecation/omission signal.
- Surfaces planned for removal should first appear in `deprecated_surfaces` or `omitted_surfaces` so KHS can fail closed or adjust activation checks before a breaking version.
