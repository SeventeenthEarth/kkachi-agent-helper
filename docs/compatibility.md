## Compatibility rules

- KHS may consume KAH `@latest` when `kkachi-agent-helper capabilities --json` reports the command surfaces and compatibility flags KHS requires. KHS release tags should also publish tested/recommended KAH versions for reproducible historical runs.
- Helper state schemas are versioned by the project-local `.kkachi/config.yaml` and embedded schema registry.
- `project init` owns KAH project bootstrap files: `.kkachi/config.yaml`, `.kkachi/project-overlay.yaml`, `docs/kkachi-docs-map.yaml`, status, events, and schema copies.
- `project init --force` reconfigures bootstrap files without deleting status, events, runs, artifacts, or gate history.
- KAH does not install KHS/Hermes skill content, templates, registries, or evaluation assets. Use Hermes native skill installation for skill packages.
- KHS owns workflow policy: whether KHS should trigger, phase applicability/order, normalized `checklist.md` generation, backend-use decisions, and commander reasoning. KAH owns deterministic state, artifact, schema, gate, event, lock, diagnostics, command-surface, phase-plan, backend-evidence, and approval validation after KHS or the user chooses to apply the workflow.
- KAH's `plan` gate requires completed `acceptance-criteria.md`, `plan.md`, and KHS-normalized `checklist.md`, but it does not parse or require KAB planner-only sections such as `KHS Checklist Seed`.
- KHS declares backend evidence requirements through `run create --backend-evidence auto|required|not_applicable`; KAH stores the resolved `backend_evidence` value, initializes required backend artifacts only when declared required, and validates artifact shape/completion without choosing or overriding the backend.
- KHS may write canonical run artifacts through `artifact write`, `artifact append`, and markdown `artifact set-status` to get KAH path-safety checks, atomic file replacement, and `artifact.written` audit events. Direct artifact file compatibility remains available during migration; this release intentionally rejects unmanaged supplemental artifact paths. Schema-owned backend JSON such as `selected-cli.json` and `bridge-session-snapshot.json` must be written with `artifact write`; generic lifecycle `set-status` is rejected with `artifact_status_not_applicable` so semantic fields like `selected-cli.json.status=supported|degraded` are preserved and `gate check backend` remains the completion validator.
- KHS may store declared workflow state in `.kkachi/runs/<run_id>/phase-plan.yaml` through `phase-plan init/show/set/validate`. KAH validates declared structure, reasons, feedback bounds, final evidence links, and approval records for rows marked `approval_required: true` only; KHS owns phase applicability, ordering policy, and workflow decisions.
- Workflow graph compatibility: `.kkachi-workflow.yaml` plus `kkachi-agent-helper graph init`, `graph validate`, `graph explain`, `graph diff`, `graph propose`, `graph apply`, `graph export`, and diagnostics `graph_compatibility` are supported when the effective KAH binary advertises those graph subcommands and compatibility flags through capabilities/help. KHS chooses templates, policy, declarative complete candidate graph proposals, and approval semantics; KAH initializes from `khs-default` or explicit template paths, validates/explains/diffs graph state, records proposal evidence, applies approved proposal records, exports non-authoritative generated diagrams, and never silently consumes fallback workflow policy. Missing, stale, broken, or otherwise invalid project graph states may be repaired only through approval-gated complete-candidate proposal/apply evidence; KAH records base validation status/reason codes, fail-closed drift checks, optional backup/recovery refs, and `graph.applied` events. `graph validate --json`, `graph explain --json`, proposal endpoint evidence, and diagnostics `graph_compatibility` expose stable `reason_codes` for implemented graph states so KAS can distinguish missing, valid, invalid-schema, parse-error, source-precedence, feedback-intake, repairability, and nested forbidden-fallback facts without KAH deciding KAS policy. Checksum/audit, manual-edit, phase-conflict, and approval-required codes are reserved vocabulary for producer-backed states and must not be treated as emitted until corresponding validation/diagnostic evidence exists. KHS must not silently edit `.kkachi-workflow.yaml` as fallback when graph apply support is missing.
- Workflow catalog promotion compatibility: `kkachi-agent-helper workflow catalog propose/apply` is supported only when the effective KAH binary advertises `workflow_catalog_proposal_apply=true` and workflow subcommands `catalog propose` and `catalog apply` through capabilities/help. KAS WFLOW-009 supplies complete candidate content and approval evidence; KAH validates candidate workflow DAG/catalog/node-contract content, records proposal evidence, enforces base/candidate checksum drift checks, verifies hash-bound approval, creates backups for existing targets, writes project-local workflow catalog targets, and appends audit events. KAH must not decide promotion policy, workflow semantic preference, selector/ranking behavior, trigger generation policy, agent assignment, backend fallback/execution, retry/rollback automation, dynamic node creation, or automatic promotion. Missing support in the installed/effective binary means KAS promotion apply must fail closed rather than direct-write `.kkachi/` workflow state.
- Declarative workflow gates are supported as graph extensions to built-in KAH gates: built-in gates win name resolution, graph-declared gates with fixed check types may be run through `gate check`, and graph gates with `final_required: true` are required and freshness-checked by `gate final`. Check types are limited to `artifact.exists`, `markdown.field`, `text.contains`, `text.contains_all`, `gitignore.contains_all`, `codegraph.evidence`, and `phase.status`. `gitignore.contains_all` lets a project opt into required ignored runtime/tool directories such as `.kkachi/`, `.codegraph/`, `.omx/`, `.omc/`, and `.claude-octopus/`; `codegraph.evidence` lets a project opt into run-local `codegraph-evidence.md` proof without making KAH execute CodeGraph itself.
- KHS should consume `kkachi-agent-helper graph explain --json` as the authoritative read-only graph projection instead of re-parsing `.kkachi-workflow.yaml` with a separate YAML parser.
- Graph source precedence must fail closed: `.kkachi/config.yaml`, generated Mermaid/PlantUML diagrams, stale `.kkachi/` runtime state, KHS defaults, and Kkachi v2 `.kkachi/config/workflows/` are never fallback graph authority. KAB graph alignment is future/non-authoritative and must not be used as policy authority.
- Configurable `EXTERNAL_FEEDBACK_INTAKE` support is implemented for graph validation/projection, phase-plan feedback-bound validation, stale-bound migration evidence, diagnostics, and capability activation checks. KAH recognizes a valid `.kkachi-workflow.yaml` `feedback_intake` declaration with `policy: "EXTERNAL_FEEDBACK_INTAKE"`, `schema_version: "external-feedback-intake/v1"`, `min_rounds: 1`, `max_rounds: 5`, `required_rounds: [1]`, and `optional_rounds: [2,3,4,5]`; stale `max3`/`1..3`, missing, duplicate, unsupported, conflicting, or round 6+ declarations fail closed. A stale-only `feedback_intake_stale_bounds` graph can be repaired only through proposal/apply evidence with a valid canonical candidate graph, no unrelated semantic graph changes, and explicit approval/audit evidence. `phase-plan validate` requires a valid graph feedback policy for declared feedback phase rows, requires paired round 1, allows optional declared rounds 2 through 5, and rejects missing graph policy instead of falling back to legacy `1..3` bounds. KHS activation must fail closed unless the effective KAH binary advertises `workflow_graph_configurable_feedback_intake=true` through `capabilities --json` and diagnostics report `graph_compatibility.feedback_intake.status=="pass"`; docs, graph help, direct YAML parsing, or partial behavior are not sufficient activation evidence.
- Token-economy evidence support is implemented for token-001 and token-002 through the `token-economy` gate, canonical `token-economy-evidence.json` artifact, embedded/exported `token-economy-evidence` schema, and the `token_economy_evidence_gate=true` / `token_economy_token002_evidence_gate=true` capability flags. KHS/KAS must supply the evidence artifact; KAH only validates deterministic shape, repository-confined refs, checksums, markers, result vocabularies, required `not_applicable` reasons, matching token task identity, and broad-mutation approval refs. For token-002, KAH additionally validates the mechanical evidence references for KAS-selected verification profile evidence, no-agent runner artifacts, token008 phase packet/run summary evidence, token009 review bundle and watcher terminal evidence, token010 change-verification matrix evidence, and mutation approval evidence. KAH does not judge English prose quality, summarize logs, choose verification profiles, manage project KAS lifecycle, activate KAB, or mutate Hermes runtime/profile/provider/auth/token/gateway/model settings.
- MAR evidence support is implemented source-side for `task_id=mar-*` or explicit run-manifest requirements through the `multi-agent-review` gate, canonical `multi-agent-review/status.json` artifact, embedded/exported `multi-agent-review-evidence` schema, final-gate integration when MAR is required, and the `multi_agent_review_evidence_gate=true` / `multi_agent_review_evidence_schema=true` capability flags. KAS/KHS must supply role-first MAR evidence from the KAS `MAR-005` contract; KAH only validates status vocabulary, passable top-level status, matching run/task identity, required role coverage records, provider attempt identity/status/failure reasons, mutation-guard fields for successful attempts, repository-confined evidence refs, checksums/markers, Blue disposition refs, required Red adjudication refs, alternate/waiver approval refs, and premium approval refs when used. KAH does not choose reviewers/providers/models, execute provider CLIs, generate prompts, adjudicate findings, approve waivers, or mutate KAB/Hermes runtime/profile/provider/auth/token/gateway/model settings.
- Policy-promotion evidence support is implemented source-side for `task_id=POLPR-007` through the `policy-promotion` gate, canonical `policy-promotion-evidence.json` artifact, embedded/exported `policy-promotion-evidence` schema (`polpr007.v1`), and the `policy_promotion_evidence_gate=true` / `policy_promotion_evidence_schema=true` capability flags. KAS/KHS must supply the evidence artifact; KAH only validates deterministic shape, repository-confined refs, checksums/markers, document impact map refs, project-Gray coverage refs, test-layer labels, failed-test repair ownership, final stale-status surfaces, boundary evidence, and required `not_applicable` reasons.
- KHS may record high-risk phase approvals with `approval request`, `approval record`, and `approval show`; KAH stores the declaration/decision and never decides when approval is required.
- KHS should prefer `kkachi-agent-helper capabilities --json` for `@latest` activation checks. The report exposes helper build info, embedded project schema version, supported command groups, compatibility flags including artifact mutation, phase-plan support, approval records, read-only workflow graph support, workflow graph init support, workflow graph apply support, workflow graph export support, workflow graph diagnostics support, workflow graph no-direct-YAML-fallback support, workflow graph configurable feedback-intake support, task-DAG schema validation support, workflow instance state support, workflow catalog diagnostics support, workflow catalog proposal/apply support, workflow final-gate integration support, KAS node-contract registry evidence support, token-economy evidence gate support, MAR evidence gate/schema support, policy-promotion evidence gate/schema support, and explicitly omitted surfaces such as the removed `install` command.
- KAH must not become the workflow-policy owner, planner, backend selector, code reviewer, KAB session controller, or Hermes skill installer; compatibility checks should fail closed when KHS requires a surface that KAH does not advertise.
- Examples and release artifacts must stay local and secret-free. Do not publish tokens, API keys, bearer headers, bridge session secrets, or production repository paths in docs, bundles, or release notes.

## Capabilities schema evolution

`capabilities_schema_version` versions the `capabilities --json` contract independently from helper release version and project state schema version.

- Additive changes that preserve existing field names, JSON types, and meanings may keep the same capabilities schema version. Examples: adding a new command group, subcommand, compatibility flag, or deprecated/omitted surface entry.
- Breaking changes must bump `capabilities_schema_version`. Examples: removing, renaming, or changing the JSON type of an existing field; changing the meaning of an existing compatibility flag; changing a previously supported command surface to unsupported without first exposing a deprecation/omission signal.
- Surfaces planned for removal should first appear in `deprecated_surfaces` or `omitted_surfaces` so KHS can fail closed or adjust activation checks before a breaking version.
